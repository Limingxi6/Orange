# 橘源通小程序计划方案

## 1.1 项目名称与定位

### 项目名称

橘源通小程序：面向华中农业大学特色柑橘产业的智能识别、追溯与估价系统

### 一句话定位

面向柑橘种植户、收购商和消费者，提供"病虫害识别 + 农事日志追溯 + 天气风险预警 + 果实分级与建议零售价"的一体化服务。

### 项目边界

一期只聚焦柑橘，并进一步建议先做脐橙/纽荷尔脐橙这一单品，避免范围过大。后续再扩展到伦晚、夏橙、杂柑。

---

## 1.2 建设目标

### 2.1 业务目标

1. 帮助农户快速识别柑橘叶片或果实上的常见问题。
2. 自动沉淀农事日志，形成可信追溯记录。
3. 结合未来 15 天天气，给出风险预警和管理建议。
4. 收获后对果实进行品质分级，并给出建议零售价区间。
5. 买家扫码查看种植过程，提升信任和议价能力。

### 2.2 软件工程目标

1. 做成一个完整系统，而不是单独的 AI demo。
2. 有明确角色、清晰模块、可维护架构、可扩展数据模型。
3. 支持后续接入更多作物、更多病害、更多市场数据。

---

## 1.3 用户角色与核心场景

### 3.1 用户角色

**农户**

- 管理地块、批次
- 拍照识别病虫害
- 记录浇水、施肥、喷药、修剪、采摘
- 收获后拍照分级、查看建议售价

**买家/消费者**

- 扫码查看批次信息
- 查看农事日志、图片证据、上链凭证
- 查看果实等级和销售价格

**管理员/专家**

- 维护病害知识库
- 维护价格基准和估价规则
- 审核识别结果和用户数据
- 配置天气预警规则

### 3.2 核心场景

1. 农户发现叶片异常，拍照后得到"疑似病害 + 建议处理方向"。
2. 识别结果自动写入该批次农业日志。
3. 系统每天同步天气并对未来 15 天做风险预警。
4. 采摘后，农户拍摄果实，系统给出等级和建议零售价区间。
5. 产品生成二维码，买家扫码查看来源和种植过程。

## 1.4 功能模块设计

### 4.1 用户与地块/批次管理

**功能：**

- 微信授权登录
- 农户信息维护
- 地块信息维护
- 作物批次创建
- 批次状态管理：种植中、预采收、已采收、已售卖

**核心数据：**

- 用户
- 地块
- 品种
- 种植批次
- 生育阶段

### 4.2 病虫害识别模块

**输入：** 柑橘叶片或果实图片

**输出：**

- 疑似问题类别
- 置信度
- 风险等级
- 简要建议
- 是否建议人工复核

**一期建议识别 5 类左右：**

- 健康
- 柑橘溃疡病
- 炭疽类病斑
- 虫害斑/叶片受害
- 日灼/机械损伤

> **注意：** 系统显示"疑似"而不是绝对诊断，避免过度承诺。

### 4.3 农业日志模块

**日志类型：**

- 拍照识别记录
- 浇水
- 施肥
- 喷药
- 修剪
- 套袋
- 采摘
- 分级
- 上架销售

**日志字段：**

- 时间
- 地块
- 批次
- 操作类型
- 图片/视频
- 操作人
- 备注
- 系统自动标签

### 4.4 天气预警模块

**功能：**

- 获取未来 15 天天气
- 结合生育阶段、历史病害记录、天气变化生成风险提醒
- 提供每日建议

**输出示例：**

- 未来 7 天病害风险：中
- 风险原因：连续高湿、降雨偏多
- 建议：减少叶面潮湿，重点巡园，必要时人工复核

> 一期建议用"规则引擎"，不要一开始就做复杂预测模型。

### 4.5 果实分级与建议零售价模块

**功能：**

- 拍照识别果实外观
- 计算品质等级
- 给出建议零售价区间
- 解释定价原因

**输出示例：**

- 品类：脐橙
- 等级：一级果
- 建议零售价：6.8～8.2 元/斤
- 理由：色泽均匀、表皮缺陷少、果径较大、追溯完整

### 4.6 区块链追溯模块

**思路：**

- 原始日志存在数据库/对象存储
- 关键节点摘要做哈希
- 哈希上链存证
- 买家端展示"可验证追溯记录"

**上链节点建议：**

- 种植批次创建
- 首次识别记录
- 喷药记录
- 采摘记录
- 分级记录
- 上架记录

### 4.7 买家端溯源展示

**内容：**

- 产品名称
- 品种与批次
- 地块信息
- 农事时间线
- 病害识别记录
- 分级与价格信息
- 上链摘要与校验状态

## 1.5 整体技术架构

> 采用"小程序前端 + 业务后端 + AI 服务 + 存证服务"的四层架构。

### 5.1 前端

**微信小程序**

- 适合农户直接使用
- 核心页面少，操作简单
- 支持扫码溯源

**建议页面：**

1. 首页
2. 地块/批次管理
3. 病虫害拍照识别
4. 农事日志
5. 天气预警
6. 果实分级与估价
7. 我的产品/二维码
8. 买家溯源页

### 5.2 管理端

**Vue3 + Element Plus**

- 用于管理员和专家配置
- 维护病害知识库、价格基准、用户信息、模型版本

### 5.3 业务后端

**Spring Boot**

适合软件工程项目，结构清晰，便于分层设计。

**分层建议：**

- controller
- service
- repository/mapper
- domain/entity
- dto/vo
- security
- scheduler

### 5.4 AI 服务

**Python + FastAPI**

负责：

- 图像识别
- 果实分级
- 风险评分
- 模型推理接口

原因：Python 方便接模型，FastAPI 接口轻量，部署方便。

### 5.5 数据与存储

## 1.6 AI 模块实施方案

### 6.1 病虫害识别

**方案**

一期采用"分类模型 + 可选定位模型"：

- 分类模型：EfficientNet / ResNet
- 定位模型：YOLOv8（后续增强）

**数据来源**

1. 自建数据集：实地拍摄叶片和果实图片
2. 公开数据集补充
3. 专家/助教标注

**标注规则**

- 一张图只标一个主标签
- 不确定样本进入"待复核"
- 模糊图直接剔除
- 记录拍摄时间、天气、地块、阶段

**指标目标**

- 分类准确率目标：85% 以上
- 高风险类别召回率优先
- 推理时间：单张 3～5 秒内

**产品策略**

- 输出"疑似"
- 高风险结果提示"建议人工复核"
- 给知识库链接，不直接给危险处方

---

### 6.2 天气风险预警

**一期方案：规则引擎**

**输入：**

- 未来 15 天天气
- 当前生育阶段
- 近 7 天日志
- 历史识别记录

**规则示例：**

- 连续高湿 + 降雨 + 前期有病斑记录 → 风险提高
- 高温暴晒 → 日灼风险提高
- 连续阴雨 + 采收临近 → 果面问题风险提高

**输出：**

- 低/中/高
- 风险解释
- 农事建议

**二期方案：轻量机器学习**

- Logistic Regression / XGBoost
- 训练目标：未来若干天内是否出现问题
- 只在数据量够用后再上

---

### 6.3 果实分级与建议零售价

**分级原则**

一期不追求专业分选设备级精度，而是做"辅助分级"。

**判断维度：**

- 色泽均匀度
- 表皮缺陷面积占比
- 果形圆整度
- 果径估计
- 成熟度

**拍摄规范**

为了估算大小，必须要求用户：

- 在固定背景板上拍摄
- 背景板带刻度或参考卡
- 单果或单层摆放
- 光照尽量均匀

> 没有参考物，系统就不要给精细尺寸，只给外观等级。

**估价逻辑**

不直接预测单个精确价格，而是输出区间。

建议零售价 = 市场基准价 × 品质系数 × 渠道系数 × 包装系数 × 追溯溢价系数

```

| 系数 | 说明 |
|------|------|
| 市场基准价 | 管理员维护 |
| 品质系数 | 根据分级结果 |
| 渠道系数 | 摆摊/批发/电商 |
| 包装系数 | 简装/礼盒 |
| 追溯溢价系数 | 完整日志可略上浮 |

**结果展示**
- 建议零售价区间
- 建议批发价区间
- 定价理由
- 风险提示：最终价格受地区和市场波动影响

## 1.7 区块链实施方案

### 7.1 为什么只做"摘要上链"

不建议把所有图片和日志全文都上链，因为：
- 性能差
- 成本高
- 演示复杂
- 没必要

### 7.2 具体流程

1. 业务系统生成日志 JSON
2. 按固定字段顺序序列化
3. 计算 SHA-256
4. 原始内容存 MySQL/MinIO
5. 哈希、批次号、时间戳、类型写入 Fabric
6. 返回交易 ID
7. 买家端显示校验结果

### 7.3 链上数据字段

| 字段 | 说明 |
|------|------|
| batch_id | 批次 ID |
| event_type | 事件类型 |
| content_hash | 内容哈希 |
| timestamp | 时间戳 |
| operator_id | 操作人 ID |
| tx_id | 交易 ID |

### 7.4 演示价值

答辩时重点讲：
- 防篡改
- 可验证
- 买家可追溯

> 不要把重点放在区块链底层原理。
```
