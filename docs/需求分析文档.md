# 橘源通项目需求分析文档

> 版本：v1.0  
> 日期：2026-03-30  
> 适用范围：微信小程序前端 + NestJS 后端

## 1. 项目概述

### 1.1 项目背景

橘源通面向柑橘种植与流通场景，目标是将生产管理、病害识别、风险预警、果品定价和溯源能力整合为一体化系统，帮助种植户和管理者提升管理效率、降低经营风险，并增强产品可信度。

### 1.2 项目目标

- 打通“种植批次 -> 农事日志 -> 病害识别 -> 风险评估 -> 产品二维码 -> 溯源查询”的闭环。
- 提供移动端高频操作入口，降低田间记录门槛。
- 提供标准化后端接口和统一数据模型，支持后续业务扩展。

### 1.3 系统范围

- 前端：微信小程序（页面、组件、服务请求封装）。
- 后端：NestJS API 服务（鉴权、业务模块、文件上传、风险引擎、溯源逻辑）。
- 数据层：MySQL（Prisma ORM）+ Redis（缓存/短期数据）。

## 2. 现状总结（As-Is）

### 2.1 前端现状

- 已配置 13 个页面，覆盖首页、登录、批次管理、农事日志、病害识别、风险预警、果实分级、二维码与溯源等核心场景。
- 已形成统一 `services` 服务层（认证、批次、日志、病害、风险、价格、产品、溯源、天气、上传）。
- 已具备统一请求封装与鉴权注入能力，支持真实接口优先 + mock 降级。

### 2.2 后端现状

- 已按领域完成模块化设计：`auth`、`batch`、`log`、`disease`、`risk`、`price`、`product`、`trace`、`weather`、`upload`、`user`。
- 已具备 JWT 鉴权、角色权限控制、DTO 参数校验、统一响应结构、Swagger 文档能力。
- 已建立核心业务数据模型：用户、批次、农事日志、病害记录、风险记录、产品、溯源记录、天气缓存。

### 2.3 关键问题

- 自动化测试体系薄弱，回归保障不足。
- 文档与部分接口实现存在命名或路径不一致风险。
- 个别前后端接口交互方式存在潜在不匹配（如病害识别上传参数格式）。
- 当前默认环境配置偏开发态，需上线前严格收敛。

## 3. 用户与角色需求

### 3.1 用户角色

- 种植户（farmer）：管理批次、记录农事、查看识别结果和风险信息。
- 管理者（manager）：查看统计数据、管控生产过程、审核关键操作。
- 管理员（admin）：全局配置、权限管理、运维治理。

### 3.2 核心诉求

- 低门槛记录：在移动端快速完成批次与日志录入。
- 可视化决策：快速识别病害和风险，获取可执行建议。
- 可信溯源：对外提供二维码查询与链路校验能力。
- 稳定可靠：关键流程在网络波动和异常场景下可恢复、可追踪。

## 4. 功能需求（To-Be）

### 4.1 认证与账号

- 支持手机号登录、验证码登录、登出、个人信息获取。
- 采用 token 持久化与过期处理机制，未登录自动引导。
- 角色权限需要覆盖接口访问与关键页面操作。

### 4.2 批次管理

- 支持批次列表查询（分页/关键字）、详情查询、创建批次、阶段更新。
- 批次需具备基础属性（品种、种植时间、地块、面积、阶段状态等）。
- 页面需覆盖 `loading/error/empty` 状态，避免静默失败。

### 4.3 农事日志

- 支持按批次查询日志、创建日志、日志类型枚举。
- 日志字段需支持文本、时间、类型、可选图片与关联批次。
- 支持下拉刷新与筛选，保障回看效率。

### 4.4 病害识别

- 支持上传图片触发识别，返回标签、置信度、严重程度、建议。
- 支持识别历史查询和按批次筛选。
- 识别结果可一键沉淀为农事日志，形成可追溯记录。

### 4.5 风险预警

- 支持风险汇总、风险评估、风险历史查询。
- 风险评估应综合天气、病害、生产行为等多源数据。
- 风险输出需要分级和建议，支持页面可视化展示。

### 4.6 果实分级与定价

- 支持果实分级/定价计算，提供基准线查询与建议能力。
- 输入参数需包含尺寸、糖度、外观等质量指标（可扩展）。
- 结果需可解释，便于业务人员理解和复核。

### 4.7 产品二维码与溯源

- 支持产品列表查询、二维码生成、二维码预览。
- 支持按溯源码查询链路信息，支持链完整性校验。
- 对外展示需控制敏感字段，保障隐私与合规。

### 4.8 天气服务与上传服务

- 支持实时天气与预报查询，为风险引擎提供输入。
- 上传服务需支持文件大小/MIME 限制和可切换存储策略。

## 5. 非功能需求

### 5.1 安全性

- 所有非公开接口默认鉴权，采用 JWT + 角色控制。
- 输入参数强校验（白名单、类型约束、字段过滤）。
- 上传能力必须校验格式和大小，并记录关键操作日志。

### 5.2 性能与稳定性

- 列表接口需支持分页和必要索引。
- 允许并发查询优化页面首屏时延。
- 天气等高频数据需缓存策略，避免外部依赖抖动放大。

### 5.3 可维护性与扩展性

- 前后端分层清晰：控制器/服务/DTO/配置分离。
- 统一错误码和响应结构，降低联调成本。
- 风险引擎和上传存储采用可替换策略，支持后续演进。

### 5.4 可观测性

- 保留请求级日志、异常日志、关键业务事件日志。
- 补充健康检查、核心指标和告警机制（后续建设）。

## 6. 约束与依赖

- 终端形态为微信小程序，需遵循平台域名与上传限制。
- 后端依赖 MySQL、Redis 与对象/本地存储服务。
- 外部 AI/算法能力与天气数据源的稳定性会影响核心流程。

## 7. 风险分析与改进建议

### 7.1 主要风险

- 缺少系统化自动化测试，版本迭代风险高。
- 开发态配置与 mock 降级策略若未收敛，可能掩盖线上问题。
- 文档和实现不一致将导致前后端联调与验收偏差。

### 7.2 改进建议

- 建立后端单测与 e2e 冒烟测试，优先覆盖认证、批次、日志、病害、溯源主链路。
- 建立接口契约校验机制（文档自动生成或契约测试）。
- 上线前冻结配置：关闭不必要调试、收敛跨域策略、禁用 mock 降级。

## 8. 里程碑建议

- M1（基础可用）：完成登录、批次、日志、病害识别主流程端到端跑通。
- M2（业务闭环）：完成风险预警、定价分级、二维码与溯源完整链路。
- M3（上线准备）：补齐测试、监控告警、配置治理与安全加固。

## 9. 验收标准（建议）

- 业务验收：核心 5 条流程可连续执行且结果可追踪。
- 技术验收：接口成功率、页面响应时延、关键错误率达到目标阈值。
- 发布验收：生产配置收敛、文档齐全、回滚与应急预案明确。
