# 1 引言

## 1.1 编写目的

本文档用于描述“橘源通”后端系统当前实现的整体设计，包括架构分层、模块职责、接口定义、数据模型和集成测试方案，作为研发、测试、运维和后续迭代的统一依据。

预期读者：
- 后端开发：理解模块边界、服务依赖与接口约束；
- 前端开发：明确接口路径、认证方式、返回结构；
- 测试工程师：据此设计集成测试和回归测试用例；
- 项目负责人/答辩成员：用于展示系统工程化设计能力。

## 1.2 范围

### 1.2.1 系统目标

系统围绕柑橘产业数字化场景，提供以下核心能力：
- 批次管理：管理果园批次信息与阶段状态；
- 农事日志：沉淀浇水、施肥、喷药、采摘等日志时间线；
- 病害识别：支持图片上传、外部 AI 推理调用和识别结果入库；
- 风险预警：结合天气、病害记录、灌溉日志进行规则评估；
- 分级定价：按规则进行果实等级判断和建议价格计算；
- 产品溯源：生成二维码和溯源码，聚合批次/日志/识别信息并校验哈希。

### 1.2.2 主要软件需求

1) 功能需求
- 用户认证与授权（JWT + 角色权限）；
- 批次的创建、列表、详情、阶段更新；
- 农事日志查询、新增、日志类型字典；
- 病害识别预测与记录查询；
- 风险摘要、风险评估、历史查询；
- 价格分级计算、基准价格、建议定价；
- 产品列表查询、二维码生成；
- 溯源查询与链上哈希校验；
- 天气当前与预报查询；
- 图片上传（本地/MinIO）。

2) 非功能需求
- 接口统一响应格式：`{ code, message, data }`；
- 全局鉴权：除公开接口外均需 JWT；
- 参数校验：全局 `ValidationPipe` + DTO 校验；
- 可观测性：统一异常过滤、标准化错误码；
- 可扩展性：AI 推理、天气服务、上传存储均采用可替换实现。

### 1.2.3 软件设计约束、限制

- 技术栈约束：NestJS + Prisma + MySQL + Redis；
- 数据库约束：当前采用 MySQL，字段类型以 MVP 兼容性优先；
- AI 依赖约束：病害识别依赖外部推理服务（`DISEASE_AI_BASE_URL`）；
- 环境约束：开发环境开放 Swagger（`/docs`），生产建议关闭；
- 接口前缀约束：默认前缀 `/api`，部分 AI/链路路径按当前代码做排除配置。

# 2 体系结构设计

## 2.1 需求复审

结合现有实现，对需求进行复审结果如下：
- 业务闭环已具备：批次 -> 日志 -> 识别 -> 风险 -> 定价 -> 产品 -> 溯源；
- 角色体系已落地：`farmer/manager/admin`，并提供角色和“本人或管理员”访问控制；
- 风险预警一期采用规则引擎，符合“先规则后模型”的实施策略；
- 病害识别采用“后端编排 + 外部 AI 调用 + 数据落库”，兼顾可用性与后续替换能力；
- 溯源采用“摘要哈希校验”实现可验证性，不直接依赖完整区块链节点。

## 2.2 软件体系结构

系统采用“表现层 + 应用层 + 领域规则 + 基础设施层 + 外部服务”架构。

**图1 橘源通后端体系架构图（逻辑）**

```text
[微信小程序 / 管理端]
          |
      HTTPS(JSON)
          |
[NestJS API 层：Controller + DTO + Guard]
          |
[应用服务层：Auth/Batch/Log/Disease/Risk/Price/Product/Trace/Weather/Upload]
          |
[领域规则层：RiskEngine / PriceEngine / TraceHash]
          |
[基础设施层：Prisma(MySQL) / Redis / Local&MinIO存储]
          |
[外部依赖：Disease AI 推理服务 / 天气 Provider]
```

架构说明：
- 接口层：负责协议、参数校验、鉴权与统一响应；
- 服务层：负责业务流程编排与事务边界；
- 规则层：沉淀可测试的规则逻辑（风险评估、价格计算、内容哈希）；
- 持久化层：通过 Prisma 统一访问数据库；
- 外部适配：通过 Client/Provider 模式接入 AI 与天气服务。

## 2.3 模块设计

| 模块 | 主要职责 | 核心接口 | 依赖 |
|---|---|---|---|
| Auth | 登录、发码、个人信息、退出 | `/api/auth/*` | Prisma、Redis、JWT |
| User | 用户信息查询、角色判定 | `/api/users/*` | Prisma、RolesGuard |
| Batch | 批次查询/创建/阶段更新 | `/api/batches/*` | Prisma |
| Log | 农事日志查询、新增、类型字典 | `/api/logs/*` | Prisma |
| Upload | 图片上传与存储适配 | `/api/upload/image` | Local/MinIO 存储 |
| Disease | 病害预测、记录查询 | `/api/disease/*` | Upload、AI Client、Prisma |

# 3 接口设计

## 3.1 用户接口

本系统后端主要面向两类前端：
- 微信小程序（农户/买家场景）：批次、日志、识别、风险、定价、溯源；
- 管理端（管理员/专家场景）：用户、价格、知识维护（当前重点为后端基础能力）。

界面交互序列（后端视角）：
1. 登录获取 token；
2. 进入批次列表和详情；
3. 记录农事日志、上传图片；
4. 发起病害识别；
5. 查询风险预警与历史；
6. 分级定价并生成产品二维码；
7. 买家端通过 traceCode 查询与验证。

## 3.2 外部接口

| 外部接口 | 交互方式 | 输入 | 输出 | 说明 |
|---|---|---|---|---|
| Disease AI 推理服务 | HTTP POST | `imageUrl`, `batchId` | `diseaseName/confidence/severity/suggestion/modelVersion/boxes` | 在 `DiseaseInferenceClient` 中封装 |
| 天气数据 Provider | 服务接口调用 | `regionCode/city` | 当前天气与 15 天预报 | 通过 `WeatherProvider` 抽象实现 |
| MinIO（可选） | SDK/API | 图片二进制 | 文件名、访问路径 | 上传模块按配置切换 |
| Redis（可选） | Key-Value | 验证码缓存 | 命中/过期 | Redis 不可用时降级内存缓存 |

说明：当前代码层面未直接接入真实区块链节点，溯源校验以链摘要哈希比对方式实现。

## 3.3 内部接口

### 3.3.1 内部模块间关系

- `DiseaseService` -> `UploadService` -> `DiseaseInferenceClient` -> `Prisma`；
- `RiskService` -> `RiskEngineService` + `Prisma(weather/disease/log)`；
- `PriceService` -> `PriceEngine(determineGrade/calculatePrice)`；
- `ProductService` -> `TraceRecord` 占位生成；
- `TraceService` -> 聚合 `Product/Batch/FarmingLog/DiseaseRecord` 并计算 hash；
- `AuthService` -> `RedisService`（失败降级内存）+ `JwtService`。

### 3.3.2 接口数据描述

#### 关键业务接口（摘录）

| 接口 | 方法 | 关键参数 | 参数说明 | 返回关键字段 |
|---|---|---|---|---|
| `/api/auth/login` | POST | `phone`, `password` | 手机号密码登录 | `token`, `userInfo` |
| `/api/batches` | GET | `keyword`, `stage`, `page`, `pageSize` | 批次查询 | `list`, `total` |
| `/api/logs` | POST | `batchId`, `type`, `content`, `operationDate` | 新增日志 | `id`, `time`, `operator` |
| `/api/disease/predict` | POST(multipart) | `file` 或 `imageUrl`, `batchId` | 病害识别 | `diseaseName`, `confidence`, `severity` |
| `/api/risk/assessment` | GET | `batchId` | 风险评估并落库 | `overallLevel`, `riskItems`, `suggestions` |
| `/api/price/grade` | POST | `diameter`, `brix`, `defectLevel`, `channel`, `packageType` | 分级和定价 | `grade`, `finalPrice`, `priceRange` |
| `/api/products/:id/qrcode` | POST | `accessUrl`, `traceCode` | 生成二维码 | `qrcodeBase64`, `traceCode` |
| `/api/trace/:code` | GET | `code` | 查询溯源聚合信息 | `productInfo`, `batchInfo`, `logTimeline` |
| `/api/trace/:code/verify` | GET | `code` | 校验 hash | `verified`, `chainHash` |
| `/api/weather/forecast` | GET | `regionCode/city`, `days` | 天气预报 | `city`, `days[]` |

#### 统一约定

- 认证：除 `auth/login`、`auth/send-code` 外默认要求 `Authorization: Bearer <token>`；
- 前缀：默认 `/api`，由全局前缀配置控制；
- 响应：全局拦截器包装为 `{ code, message, data }`；
- 错误：业务异常通过统一错误码与消息返回。

# 4 数据设计

## 4.1 数据结构

核心实体（按当前 `schema.prisma`）：
- `User`：用户基础信息、角色、状态；
- `Batch`：种植批次、品种、阶段、负责人；
- `FarmingLog`：农事操作日志（含图片与明细 JSON）；
- `DiseaseRecord`：病害识别记录（含原始 AI 结果 JSON）；
- `RiskRecord`：风险评估记录（含规则引擎源数据 JSON）；
- `Product`：产品与上架状态、二维码链接；
- `TraceRecord`：溯源快照、内容哈希、校验状态；
- `WeatherCache`：天气当前/预报缓存。

关键枚举：
- 用户：`UserRole(farmer/manager/admin)`, `UserStatus(active/disabled)`；
- 病害：`DiseaseSeverity(low/mid/high)`, `DiseaseStatus(normal/review)`；
- 风险：`RiskLevel(low/mid/high)`；
- 产品：`ProductStatus(pending/listed)`。

## 4.2 文件和数据库结构

### 4.2.1 数据库逻辑结构（核心表）

| 表名 | 主键 | 关键字段 | 说明 |
|---|---|---|---|
| `users` | `id` | `phone`, `password_hash`, `role`, `status` | 用户与权限 |
| `batches` | `id` | `batch_no`, `orchard_name`, `variety`, `stage`, `manager_id` | 种植批次 |
| `farming_logs` | `id` | `batch_id`, `type`, `content`, `operator_id`, `operation_date`, `images` | 农事日志 |
| `disease_records` | `id` | `batch_id`, `disease_name`, `confidence`, `severity`, `status`, `raw_result` | 病害识别结果 |
| `risk_records` | `id` | `batch_id`, `risk_type`, `risk_level`, `summary`, `suggestion`, `source_data` | 风险评估结果 |
| `products` | `id` | `batch_id`, `product_name`, `grade`, `price`, `qr_code_url`, `status` | 产品管理 |
| `trace_records` | `id` | `product_id`, `trace_code`, `batch_snapshot`, `logs_snapshot`, `inspections_snapshot`, `chain_hash`, `verified` | 溯源快照与校验 |
| `weather_cache` | `id` | `region_code`, `weather_date`, `current_data`, `forecast_data` | 天气缓存 |

### 4.2.2 关系与索引说明

- 关系：
  - `User` 1:N `Batch`（负责人）；
  - `Batch` 1:N `FarmingLog` / `DiseaseRecord` / `RiskRecord` / `Product`；
  - `Product` 1:N `TraceRecord`（按 traceCode 唯一）；
- 唯一约束：
  - `users.phone`，`batches.batch_no`，`trace_records.trace_code`；
  - `weather_cache(region_code, weather_date)`；
- 常用索引：
  - `risk_records(batch_id, created_at)`；
  - `disease_records(batch_id, created_at)`；
  - `products(batch_id, status)`。

### 4.2.3 文件存储结构

- 上传图片默认走本地目录（静态映射 `/uploads/*`）；
- 配置 `upload.mode=minio` 时，切换 MinIO 存储；
- `FarmingLog.images`、`DiseaseRecord.imageUrl` 保存文件访问路径。

# 6 测试部分

本章对应系统集成测试，重点覆盖跨模块业务链路与异常分支。

## 6.1 模块集成策略

- 主链路优先：认证 -> 批次 -> 日志 -> 识别 -> 风险 -> 定价 -> 溯源；
- 外部依赖分层测试：先 Mock AI/天气，再联调真实服务；
- 数据一致性验证：重点检查日志、识别、风险与溯源快照一致性；
- 回归策略：按模块分组执行 API 回归，关键接口每日冒烟。

## 6.2 集成测试方案

| 用例编号 | 测试目标 | 前置条件 | 步骤 | 预期结果 |
|---|---|---|---|---|
| IT-001 | 登录并访问受保护接口 | 存在有效用户 | 调 `/api/auth/login` 后访问 `/api/users/me` | 返回 token 且用户信息正确 |
| IT-002 | 批次创建与详情统计 | 登录状态 | 创建批次，新增日志和产品，再查详情 | `logsCount/productsCount` 正确 |
| IT-003 | 病害识别闭环 | 批次存在，AI 服务可达 | 上传图片调用 `/api/disease/predict` | 返回识别结果并落库 |
| IT-004 | 高风险自动复核标记 | AI 返回 `severity=high` | 触发预测后查记录 | `status=review` |
| IT-005 | 风险评估规则命中 | 有天气与病害数据 | 调 `/api/risk/assessment` | 返回 `riskItems`，并写入 `risk_records` |
| IT-006 | 定价规则正确性 | 登录状态 | 传 A/B/C 边界参数调用 `/api/price/grade` | `grade` 与 `finalPrice` 符合规则 |
| IT-007 | 产品二维码与溯源联动 | 产品存在 | 调 `/api/products/:id/qrcode` 后查 `/api/trace/:code` | 可查询并返回聚合信息 |
| IT-008 | 溯源验真 | 已有 trace record | 调 `/api/trace/:code/verify` | 返回 `verified=true`（未篡改） |
| IT-009 | 天气缓存命中 | 同区域同日重复查询 | 连续调用 `/api/weather/current` | 首次写缓存，后续命中缓存 |
| IT-010 | Redis 降级能力 | Redis 不可用 | 调 `/api/auth/send-code` | 仍可返回验证码（开发环境含 debugCode） |

## 6.3 关键风险与缓解

- 外部 AI 不可用：接口超时/非 2xx 时返回明确错误，建议增加熔断与重试策略；
- 天气数据质量波动：已支持缓存与 Provider 抽象，建议增加多源容灾；
- 规则可维护性：风险与定价规则已独立在 engine 中，建议补齐单元测试；
- 溯源可信度边界：当前为哈希校验，后续可扩展真实链上锚定与交易回查。

## 6.4 验收指标建议

- 核心链路可用率 >= 99%；
- 关键接口 95 分位响应 < 800ms（不含外部 AI 推理耗时）；
- 主要模块均有集成测试覆盖，异常场景覆盖率 >= 80%；
- 需求交叉索引中的 FR-001 ~ FR-009 均有对应可执行测试用例。
